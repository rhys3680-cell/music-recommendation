# 12. Cold Start 문제

## Cold Start란?

새로운 사용자 또는 새로운 아이템에 대한 **데이터가 없어서 추천을 할 수 없는 상황**이다.

```
추천 시스템의 전제:
  "과거 데이터를 기반으로 선호도를 예측한다"

Cold Start:
  과거 데이터가 없다 → 예측 불가
```

---

## Cold Start의 세 가지 유형

### 1. 새 사용자 (New User Cold Start)

```
신규 가입자가 들어왔다
→ 이력이 없으므로 CF 불가
→ 프로필이 없으므로 CBF 불가
→ 뭘 추천해야 하는가?
```

### 2. 새 아이템 (New Item Cold Start)

```
새 영화가 개봉했다
→ 아무도 평가하지 않았으므로 CF에서 추천 대상이 안 됨
→ 신곡, 신상품도 동일한 문제
```

### 3. 새 시스템 (New System Cold Start)

```
서비스를 처음 런칭했다
→ 사용자도 없고, 상호작용 데이터도 없다
→ 모든 것이 Cold
```

---

## 새 사용자 해결 전략

### 1. 인기도 기반 추천

가장 단순한 방법. 데이터가 없으면 **모두에게 잘 먹히는 것**을 보여준다.

```
신규 유저 → 인기 차트 Top 20 추천

장점: 구현 간단, 실패 확률 낮음
단점: 개인화 없음, 모든 신규 유저가 같은 추천을 받음
```

### 2. 온보딩 (취향 조사)

가입 시 **명시적으로 취향을 물어본다**.

```
Spotify:  "좋아하는 아티스트 3명 이상 선택하세요"
Netflix:  "좋아하는 작품을 골라주세요"
Pinterest: "관심 있는 주제를 선택하세요"

→ 선택한 것으로 초기 사용자 프로필 생성
→ CBF 또는 간단한 CF 시작 가능
```

```
설계 포인트:
  너무 많이 물으면 → 가입 이탈률 증가
  너무 적게 물으면 → 프로필 정보 부족
  일반적으로 3~10개 선택이 적당
```

### 3. 인구통계 기반

나이, 성별, 지역 등 **기본 정보**로 유사 그룹을 찾아 추천한다.

```
신규 유저: 25세 남성, 서울
→ 기존 유저 중 비슷한 프로필의 사용자들이 좋아한 것 추천

장점: 최소한의 정보로 개인화 시작
단점: 스테레오타입 추천 위험, 개인 차이 반영 못함
```

### 4. 소셜 정보 활용

SNS 로그인으로 **관심사, 친구 관계**를 가져온다.

```
Facebook 로그인 → 좋아요 페이지, 관심사 수집
→ 비슷한 관심사를 가진 기존 유저의 추천 활용

장점: 풍부한 초기 프로필
단점: 개인정보 이슈, SNS 연동 거부하는 유저 많음
```

### 5. Exploration (탐색)

초기에 **다양한 아이템을 일부러 노출**하여 빠르게 반응 데이터를 수집한다.

```
신규 유저에게:
  인기 아이템 50% + 다양한 장르/카테고리 50%

→ 클릭/스킵 반응을 빠르게 수집
→ 몇 번의 상호작용 후 개인화 시작

Multi-Armed Bandit (MAB) 활용:
  여러 "팔"(추천 전략)을 시도하면서
  반응이 좋은 쪽으로 점점 집중
```

### 6. 단계적 전환 (Progressive Personalization)

데이터가 쌓이는 만큼 **점진적으로 개인화 수준을 높인다**.

```
상호작용 0개:   인기도 기반
상호작용 1~5개: 인구통계 + 온보딩 기반
상호작용 5~20개: 간단한 CBF
상호작용 20개~:  CF / MF / 딥러닝

→ Switching 하이브리드의 실전 적용
```

## 새 아이템 해결 전략

### 1. 콘텐츠 기반 필터링

아이템의 **속성(메타데이터)**만으로 기존 아이템과의 유사도를 계산한다.

```
신곡 등록:
  장르: 팝
  아티스트: A
  BPM: 120
  분위기: 밝음

→ 기존 곡 중 비슷한 속성의 곡을 좋아한 유저에게 추천
→ 상호작용 데이터 없이도 가능
```

### 2. 메타데이터 임베딩

아이템 속성을 **잠재 벡터 공간에 매핑**하여 CF와 연결한다.

```
학습 단계:
  기존 아이템의 속성 → 신경망 → 잠재 벡터
  이 잠재 벡터가 CF에서 학습된 아이템 벡터와 일치하도록 학습

추론 단계:
  신규 아이템의 속성 → 같은 신경망 → 잠재 벡터 생성
  → CF 프레임워크에 바로 투입 가능
```

```
예시:
  기존 곡들로 학습:
    곡 속성 [팝, BPM120, 밝음] → 잠재 벡터 [0.3, 0.8, 0.5]
    
  신곡 속성 [팝, BPM125, 밝음] → 잠재 벡터 [0.3, 0.7, 0.5] (유사)
  → 기존 CF 모델에서 이 벡터로 바로 추천 가능
```

### 3. 전략적 노출 (Explore-Exploit)

새 아이템을 **일정 비율로 노출**하여 빠르게 데이터를 수집한다.

```
추천 목록 구성:
  80%: 기존 모델의 추천 (Exploit)
  20%: 새 아이템 노출 (Explore)

→ 새 아이템에 대한 반응 수집
→ 충분한 데이터가 쌓이면 정상 추천 루트로 합류

문제: 품질이 낮은 새 아이템도 노출됨
해결: CBF 점수가 일정 이상인 새 아이템만 노출 대상으로
```

### 4. Cross-Domain Transfer

다른 도메인의 데이터를 활용한다.

```
음악 서비스에서 새 아티스트가 곡을 등록
→ 이 아티스트의 YouTube 조회수, SNS 팔로워, 유사 아티스트 관계 활용
→ 다른 도메인의 인기/유사도를 초기 시그널로 사용
```

## 새 시스템 해결 전략

```
1단계: 인기도/편집자 추천으로 시작
  → 에디터가 큐레이션한 목록 노출
  → 외부 인기 차트 데이터 활용

2단계: 빠른 피드백 수집 설계
  → 좋아요/싫어요 UI를 눈에 띄게 배치
  → 온보딩 프로세스 강화
  → 암시적 피드백 (클릭, 체류 시간) 즉시 수집

3단계: 간단한 모델부터 시작
  → 인기도 → CBF → Item-Based CF → MF → 딥러닝
  → 데이터가 쌓이는 속도에 맞춰 모델 업그레이드

4단계: 외부 데이터 활용
  → 공개 데이터셋으로 사전 학습
  → Transfer Learning으로 자사 데이터에 미세 조정
```

## Cold Start 해결 전략 요약

| 유형 | 전략 | 필요한 것 |
|------|------|----------|
| 새 사용자 | 인기도 기반 | 없음 |
| 새 사용자 | 온보딩 | 취향 조사 UI |
| 새 사용자 | 인구통계 | 기본 프로필 |
| 새 사용자 | 탐색 (MAB) | 실시간 피드백 수집 |
| 새 아이템 | CBF | 아이템 메타데이터 |
| 새 아이템 | 메타데이터 임베딩 | 학습된 매핑 모델 |
| 새 아이템 | 전략적 노출 | 노출 정책 설계 |
| 새 시스템 | 단계적 전환 | 시간과 데이터 축적 |

---

## 정리

```
Cold Start = 데이터 없이 추천해야 하는 상황

세 가지 유형:
  새 사용자: 이력 없음 → 온보딩, 인기도, 탐색으로 초기 데이터 확보
  새 아이템: 평가 없음 → CBF, 메타데이터 임베딩으로 기존 체계에 합류
  새 시스템: 모두 없음 → 단계적으로 모델 고도화

핵심 원칙:
  1. 완벽한 해결은 없다 — 데이터가 쌓일 때까지의 "버팀" 전략
  2. 가능한 빨리 피드백을 수집하는 설계가 중요
  3. 단계적 전환: 인기도 → CBF → CF → 딥러닝
  4. Explore-Exploit 균형이 핵심
```
