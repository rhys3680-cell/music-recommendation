# 15. 시스템 설계 (System Design)

## 전체 그림

지금까지 배운 모든 것을 하나의 시스템으로 엮는다.

```
추천 시스템 = 데이터 → 학습 → 서빙 → 평가 → 개선의 순환

┌─────────────────────────────────────────────────────────┐
│                                                          │
│   데이터 수집 → 데이터 처리 → 모델 학습 → 서빙 → 평가   │
│       ↑                                          │       │
│       └──────────── 피드백 루프 ──────────────────┘       │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

---

## 1. 데이터 계층

### 데이터 수집

```
명시적 피드백:
  - 평점, 좋아요, 싫어요, 리뷰

암시적 피드백:
  - 클릭, 조회, 재생, 스킵, 구매
  - 체류 시간, 스크롤 깊이
  - 검색 쿼리

컨텍스트:
  - 타임스탬프, 디바이스, 위치
  - 세션 정보

아이템 메타데이터:
  - 카테고리, 태그, 설명, 이미지
  - 음악: 오디오 특성, 가사, 아티스트 정보
```

### 데이터 저장

```
실시간 데이터:
  이벤트 스트림 (Kafka, Kinesis)
  → 클릭, 재생 등 실시간 이벤트 수집

배치 데이터:
  데이터 웨어하우스 (BigQuery, Snowflake, HDFS)
  → 전체 이력, 모델 학습용

피처 저장소 (Feature Store):
  Redis, DynamoDB 등
  → 서빙 시점에 빠르게 조회할 피처
  → 유저의 최근 이력, 아이템 인기도 등
```

### 데이터 파이프라인

```
                    실시간 경로
이벤트 발생 ───→ Kafka ───→ 실시간 피처 업데이트 ───→ Feature Store
                  │
                  ↓          배치 경로
              Data Lake ───→ ETL/ELT ───→ 학습 데이터셋
                                          모델 학습
```

## 2. 피처 엔지니어링

### 사용자 피처

```
정적 피처 (잘 안 변함):
  - 나이, 성별, 국가
  - 가입일, 구독 유형

동적 피처 (자주 변함):
  - 최근 7일 클릭한 아이템 ID 목록
  - 최근 30일 장르별 소비 비율
  - 평균 세션 길이
  - 마지막 접속 시간
  - 활동 시간대 분포
```

### 아이템 피처

```
정적 피처:
  - 카테고리, 장르, 태그
  - 출시일, 제작자/아티스트
  - 길이 (영상 길이, 곡 길이)

동적 피처:
  - 최근 7일 클릭 수, 재생 수
  - 전체 평균 평점
  - 트렌딩 점수 (최근 인기 상승률)
  - CTR (노출 대비 클릭률)
```

### 교차 피처

```
유저 × 아이템:
  - 이 유저가 이 장르를 과거에 몇 번 소비했는가
  - 이 유저가 이 아티스트의 다른 곡을 들었는가
  - 이 유저의 임베딩과 이 아이템 임베딩의 유사도

유저 × 컨텍스트:
  - 이 유저가 이 시간대에 주로 뭘 소비하는가
  - 이 유저가 이 디바이스에서 선호하는 콘텐츠 유형
```

### 피처 관리

```
Feature Store의 역할:

1. 일관성: 학습과 서빙에서 같은 피처 사용
   → 학습 시 "최근 7일 클릭" = 서빙 시 "최근 7일 클릭" (동일 로직)

2. 재사용: 여러 모델이 같은 피처 공유
   → 추천 모델, 검색 모델, 광고 모델이 같은 유저 피처 사용

3. 신선도: 실시간 피처 업데이트
   → 방금 클릭한 아이템이 즉시 피처에 반영

도구: Feast, Tecton, 자체 구축
```

## 3. 모델 계층

### 오프라인 학습 파이프라인

```
학습 데이터 준비
     ↓
모델 학습 (GPU 클러스터)
     ↓
오프라인 평가 (NDCG, Recall 등)
     ↓
  통과?  ──No──→ 모델 튜닝/디버깅
     │
    Yes
     ↓
모델 배포 (Model Registry)
     ↓
A/B 테스트 배정
```

### 모델 구성 (실무 표준)

```
후보 생성 모델 (여러 개 병렬):
  ├── Two-Tower CF 모델
  ├── Item-Based 유사도 모델
  ├── 시퀀스 모델 (최근 이력 기반)
  ├── 인기도 모델
  └── 탐색 모델 (Exploration)

랭킹 모델:
  └── DeepFM / Wide&Deep / Transformer
      입력: 유저 피처 + 아이템 피처 + 교차 피처 + 컨텍스트

재정렬 로직:
  ├── 다양성 (MMR)
  ├── 비즈니스 규칙
  ├── 필터링 (이미 본 것, 부적절 콘텐츠)
  └── 공정성 보정
```

## 4. 서빙 계층

### 요청 처리 흐름

```
사용자 요청 (추천 페이지 로드)
     ↓                              지연 시간 목표
┌─────────────────┐
│ 1. 유저 피처 조회 │  ← Feature Store    ~5ms
└────────┬────────┘
         ↓
┌─────────────────┐
│ 2. 후보 생성     │  ← ANN 검색 + 규칙   ~10ms
│    (여러 소스)   │
└────────┬────────┘
         ↓
┌─────────────────┐
│ 3. 피처 조합     │  ← 후보별 피처 조회   ~10ms
└────────┬────────┘
         ↓
┌─────────────────┐
│ 4. 랭킹 모델     │  ← 모델 추론         ~20ms
└────────┬────────┘
         ↓
┌─────────────────┐
│ 5. 재정렬        │  ← 다양성, 규칙      ~5ms
└────────┬────────┘
         ↓
응답 반환 (추천 목록)               총: ~50ms
```

### 서빙 인프라

```
모델 서빙:
  TensorFlow Serving, TorchServe, Triton
  → GPU 추론 or CPU 추론 (모델 크기에 따라)

캐싱:
  유저별 추천 결과 캐싱 (Redis)
  → 같은 유저가 짧은 시간 내 재요청 시 캐시 반환
  → TTL: 수 분 ~ 수십 분

로드 밸런싱:
  여러 서빙 인스턴스에 요청 분산
  → 트래픽 피크 대응
```

## 5. 평가 계층

### 오프라인 평가

```
과거 데이터로 모델 성능 측정

방법:
  1. 시간순 분할 (가장 현실적)
     학습: ~3월 31일 데이터
     평가: 4월 1일~7일 데이터

  2. 사용자 분할
     학습: 80% 사용자 데이터
     평가: 20% 사용자 데이터

지표:
  Recall@K, NDCG@K, Hit Rate
  → 모델 후보 선별에 사용
```

### 온라인 평가 (A/B 테스트)

```
실제 사용자에게 두 가지 모델을 동시에 노출

그룹 A (대조군): 기존 모델    → 50% 트래픽
그룹 B (실험군): 새 모델      → 50% 트래픽

비교 지표:
  비즈니스 지표:
    - CTR (클릭률)
    - 체류 시간
    - 전환율 (구매, 재생 완료)
    - 재방문율

  추천 지표:
    - 추천 다양성
    - 커버리지
    - 사용자 만족도 (설문)

기간: 최소 1~2주 (통계적 유의성 확보)
```

### A/B 테스트 주의사항

```
1. 통계적 유의성
   → p-value < 0.05 확인
   → 충분한 샘플 크기 확보

2. 참신성 효과 (Novelty Effect)
   → 새 모델이 처음엔 좋아 보이지만 시간이 지나면 원래 수준
   → 충분한 기간 관찰 필요

3. 장기 효과
   → 단기 CTR은 높지만 장기 이탈률도 높을 수 있음
   → 필터 버블로 인한 장기 영향 모니터링

4. 상호작용 효과
   → 여러 A/B 테스트 동시 진행 시 간섭 가능
   → 실험 간 독립성 확인
```

### 오프라인 vs 온라인 평가

```
오프라인 NDCG 높음 → 온라인 CTR도 높을까?

항상 그렇지는 않다:
  - 오프라인은 과거 데이터 기반 (분포 변화 미반영)
  - 노출 편향: 과거에 노출된 아이템만 평가 가능
  - 비즈니스 지표와 추천 지표는 다를 수 있음

→ 오프라인 평가는 "나쁜 모델 걸러내기"에 유용
→ 최종 판단은 반드시 A/B 테스트로
```

## 6. 피드백 루프와 지속 개선

### 실시간 피드백 반영

```
유저가 추천 결과에 반응
  ↓
이벤트 수집 (클릭, 스킵, 구매)
  ↓
실시간 피처 업데이트
  ↓
다음 추천에 즉시 반영

예시:
  유저가 재즈 곡 클릭 → "최근 클릭 장르" 피처에 재즈 추가
  → 다음 추천 시 재즈 비중 증가
```

### 모델 재학습 주기

```
실시간 업데이트:  피처만 업데이트 (모델은 그대로)
일간 재학습:     매일 새 데이터로 모델 학습
주간 재학습:     큰 구조 변경 시
수시:           성능 저하 감지 시 긴급 재학습

균형:
  너무 자주 → 불안정, 비용 높음
  너무 드물게 → 트렌드 반영 못함
  일반적으로 일간~주간이 적당
```

### 모니터링

```
서비스 지표:
  - 응답 시간 (p50, p95, p99)
  - 에러율
  - 처리량 (QPS)

모델 지표:
  - 온라인 CTR, 전환율 추이
  - 추천 다양성 변화
  - 커버리지 변화

데이터 지표:
  - 데이터 파이프라인 지연
  - 피처 누락률
  - 데이터 분포 변화 (드리프트)

알림:
  - CTR이 전날 대비 20% 이상 하락 → 알림
  - 응답 시간 p99가 200ms 초과 → 알림
  - 피처 누락률 5% 초과 → 알림
```

## 전체 아키텍처 종합

```
┌──────────────────────────────────────────────────────────┐
│                     데이터 계층                           │
│  이벤트 스트림 → Kafka → Feature Store (실시간)           │
│                    └──→ Data Lake → ETL (배치)           │
└──────────────────────────┬───────────────────────────────┘
                           ↓
┌──────────────────────────────────────────────────────────┐
│                     학습 계층                             │
│  학습 데이터 → 후보 생성 모델 학습                        │
│             → 랭킹 모델 학습                             │
│             → 아이템 임베딩 계산 → ANN 인덱스 구축        │
│             → 오프라인 평가 → Model Registry 배포         │
└──────────────────────────┬───────────────────────────────┘
                           ↓
┌──────────────────────────────────────────────────────────┐
│                     서빙 계층                             │
│  요청 → 유저 피처 조회 → 후보 생성 (ANN + 다중 소스)     │
│      → 피처 조합 → 랭킹 모델 추론 → 재정렬 → 응답       │
└──────────────────────────┬───────────────────────────────┘
                           ↓
┌──────────────────────────────────────────────────────────┐
│                     평가 계층                             │
│  A/B 테스트 → 비즈니스 지표 비교 → 승자 모델 결정        │
│  모니터링 → 이상 감지 → 알림/롤백                        │
└──────────────────────────────────────────────────────────┘
```

## 단계별 구축 가이드

작은 서비스에서 시작하여 점진적으로 고도화하는 로드맵이다.

```
1단계: MVP (유저 수천 명)
  - 인기도 기반 + 간단한 규칙
  - 단일 서버, 배치 계산
  - 데이터 수집 체계 구축 (이게 가장 중요!)

2단계: 개인화 시작 (유저 수만 명)
  - Item-Based CF 또는 간단한 MF
  - 오프라인 배치 추천 (미리 계산)
  - 기본 A/B 테스트 도입

3단계: 고도화 (유저 수십만 명)
  - Two-Stage (후보 생성 + 랭킹)
  - 딥러닝 랭킹 모델
  - Feature Store 도입
  - ANN 인덱스 도입

4단계: 대규모 (유저 수백만 명+)
  - 다중 후보 생성 소스
  - 실시간 피처 업데이트
  - 분산 학습
  - 고도화된 A/B 테스트 플랫폼
  - 모니터링/알림 체계
```

## 정리

```
추천 시스템 설계 = 데이터 → 학습 → 서빙 → 평가의 순환 구조

핵심 구성:
  데이터: 이벤트 수집 → Feature Store (실시간 + 배치)
  학습: 후보 생성 모델 + 랭킹 모델 (오프라인 학습)
  서빙: 후보 생성 → 랭킹 → 재정렬 (50ms 이내)
  평가: 오프라인 지표 + A/B 테스트 (최종 판단)

실무 원칙:
  1. 데이터 수집이 가장 중요 — 모델보다 데이터가 먼저
  2. 단순한 것부터 시작 — 인기도 → CF → MF → 딥러닝
  3. 측정 없이 개선 없다 — A/B 테스트는 필수
  4. 피처가 모델보다 중요 — 좋은 피처 > 복잡한 모델
  5. 서비스 안정성 — 느린 추천보다 빠른 인기도가 낫다
```
