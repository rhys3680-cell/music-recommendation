# 13. 확장성 (Scalability)

## 왜 확장성이 문제인가?

```
소규모:
  유저 1,000명 × 아이템 1,000개 = 100만 쌍
  → 전수 계산 가능

실제 서비스:
  Spotify: 유저 6억 × 곡 1억 = 6경 쌍
  YouTube: 유저 25억 × 영상 8억 = ...
  → 전수 계산 불가능

문제:
  1. 모든 아이템의 점수를 계산할 수 없다
  2. 실시간(수십 ms 이내)으로 응답해야 한다
  3. 모델 학습도 합리적 시간 안에 끝나야 한다
```

---

## 해결 전략: 후보 생성 + 랭킹 (Two-Stage)

대규모 추천 시스템의 **표준 아키텍처**이다.

```
전체 아이템 (수억 개)
       ↓
┌──────────────────┐
│  후보 생성        │  수억 → 수백~수천 개로 축소
│  (Candidate Gen)  │  빠르지만 대략적인 모델
└──────────────────┘
       ↓
┌──────────────────┐
│  랭킹             │  수백~수천 개를 정밀 순위 매김
│  (Ranking)        │  느리지만 정확한 모델
└──────────────────┘
       ↓
┌──────────────────┐
│  필터링/재정렬     │  비즈니스 규칙, 다양성 조정
│  (Re-ranking)     │
└──────────────────┘
       ↓
  최종 추천 (10~50개)
```

### 각 단계의 특성

| 단계 | 입력 크기 | 출력 크기 | 속도 요구 | 모델 복잡도 |
|------|----------|----------|----------|-----------|
| 후보 생성 | 수억 | 수백~수천 | **매우 빠름** | 단순 |
| 랭킹 | 수백~수천 | 수십~수백 | 빠름 | **복잡** |
| 재정렬 | 수십~수백 | 10~50 | 보통 | 규칙 기반 |

## 후보 생성 방법

### 1. Two-Tower Model

사용자와 아이템을 **각각 독립된 신경망(타워)**으로 인코딩한다.

```
  유저 피처               아이템 피처
  [ID, 나이, 이력...]     [ID, 장르, 속성...]
       ↓                       ↓
  ┌─────────┐            ┌─────────┐
  │ User    │            │ Item    │
  │ Tower   │            │ Tower   │
  │ (DNN)   │            │ (DNN)   │
  └─────────┘            └─────────┘
       ↓                       ↓
  유저 벡터 (128d)        아이템 벡터 (128d)
       \                     /
        \                   /
         내적 또는 코사인 유사도
               ↓
           유사도 점수
```

#### 왜 Two-Tower인가?

```
핵심: 유저와 아이템 벡터가 독립적으로 계산됨

→ 아이템 벡터는 미리 전부 계산해서 저장 가능 (오프라인)
→ 추천 시점에는 유저 벡터만 계산하면 됨
→ 유저 벡터와 가장 가까운 아이템 벡터를 빠르게 찾으면 끝
```

### 2. 다중 후보 생성

하나의 방법만 쓰지 않고 **여러 소스**에서 후보를 모은다.

```
후보 소스 1: CF 기반 (비슷한 유저가 좋아한 것)      → 200개
후보 소스 2: Item-Based (최근 본 것과 비슷한 것)    → 200개
후보 소스 3: 인기도 기반 (트렌딩)                    → 100개
후보 소스 4: 개인화 탐색 (새로운 장르)               → 50개
후보 소스 5: 컨텍스트 기반 (시간대, 날씨)            → 50개
─────────────────────────────────────────────────
합계 (중복 제거 후):                                → 약 500개

→ 500개를 랭킹 모델에 넘김
```

## ANN (Approximate Nearest Neighbor)

후보 생성의 핵심 기술. 수억 개 벡터에서 **가장 가까운 K개를 빠르게** 찾는다.

### 정확한 검색 vs 근사 검색

```
정확한 검색 (Brute-Force):
  유저 벡터와 1억 개 아이템 벡터를 전부 비교
  → O(n) = 1억 번 내적 계산
  → 너무 느림

근사 검색 (ANN):
  100% 정확하지는 않지만 99%+ 정확도로 수천 배 빠르게
  → O(log n) 또는 O(1)에 가깝게
```

### 주요 ANN 알고리즘

#### LSH (Locality-Sensitive Hashing)

```
원리: 비슷한 벡터가 같은 해시 버킷에 들어가도록 해시 함수 설계

1. 여러 해시 함수로 벡터를 해싱
2. 같은 버킷에 있는 벡터들만 비교

         해시1  해시2  해시3 → 버킷
벡터A:    01     10     11  → 버킷 "011011"
벡터B:    01     10     11  → 버킷 "011011"  ← 같은 버킷! 유사할 확률 높음
벡터C:    10     01     00  → 버킷 "100100"

→ A와 B만 비교하면 됨 (C는 스킵)
```

#### HNSW (Hierarchical Navigable Small World)

```
원리: 그래프 기반. 벡터들을 계층적 그래프로 연결

상위 계층: 소수의 노드, 큰 점프 (고속도로)
하위 계층: 많은 노드, 작은 점프 (일반 도로)

검색 과정:
  상위 계층에서 대략적 위치 찾기 (큰 점프)
     ↓
  하위 계층으로 내려가며 정밀 탐색 (작은 점프)

→ 현재 가장 널리 쓰이는 ANN 방법
→ 높은 정확도 + 빠른 검색 속도
```

#### IVF (Inverted File Index)

```
원리: 벡터 공간을 여러 클러스터로 나누고, 가까운 클러스터만 검색

1. K-means로 벡터를 클러스터링 (오프라인)
2. 쿼리 벡터와 가장 가까운 클러스터 n개 선택
3. 선택된 클러스터 내 벡터만 비교

전체 1억 → 1000개 클러스터 → 상위 10개 클러스터 내 100만 개만 비교
→ 100배 빨라짐
```

### ANN 라이브러리

| 라이브러리 | 개발 | 특징 |
|-----------|------|------|
| FAISS | Meta | GPU 지원, 대규모에 강함, IVF+PQ |
| Annoy | Spotify | 메모리 효율, 읽기 최적화 |
| ScaNN | Google | 높은 정확도, 양자화 기법 |
| Milvus | 오픈소스 | 벡터 DB, 분산 지원 |
| Pinecone | 클라우드 | 매니지드 서비스 |

```
실무 선택 기준:
  데이터 크기 1억 이하 + GPU 있음 → FAISS
  메모리 제약 있음 → Annoy
  관리 부담 줄이고 싶음 → Pinecone, Milvus
```

## 랭킹 단계

후보 수백 개에 대해 **정밀한 점수**를 매긴다. 복잡한 모델 사용 가능.

```
후보 생성:
  유저 벡터 · 아이템 벡터 = 점수 (단순 내적)

랭킹:
  입력: 유저 피처 + 아이템 피처 + 유저×아이템 교차 피처 + 컨텍스트
  모델: Wide & Deep, DeepFM, Transformer 등
  출력: 클릭/구매/재생 확률

→ 후보 생성에서는 못 쓰는 복잡한 피처를 여기서 활용
```

### 후보 생성 vs 랭킹에서 쓰는 피처

```
후보 생성 (빠르게):
  유저 임베딩, 아이템 임베딩만 사용
  → 교차 피처 사용 불가 (미리 계산해야 하므로)

랭킹 (정밀하게):
  유저 피처: 나이, 성별, 최근 이력, 활동 패턴
  아이템 피처: 장르, 인기도, 신선도, 길이
  교차 피처: "이 유저가 이 장르를 얼마나 봤는가"
  컨텍스트: 시간대, 디바이스, 위치
  
→ 수백 개에 대해서만 계산하므로 복잡해도 괜찮음
```

## 재정렬 (Re-ranking)

랭킹 결과에 **비즈니스 로직과 다양성**을 적용한다.

```
랭킹 결과:
  1. 액션 영화A (0.95)
  2. 액션 영화B (0.93)
  3. 액션 영화C (0.91)
  4. 로맨스 영화D (0.90)
  5. 액션 영화E (0.89)

재정렬 후 (다양성 적용):
  1. 액션 영화A (0.95)
  2. 로맨스 영화D (0.90)  ← 장르 다양성을 위해 올림
  3. 액션 영화B (0.93)
  4. 다큐 영화F (0.85)    ← 탐색 목적으로 삽입
  5. 액션 영화C (0.91)
```

```
재정렬에서 처리하는 것들:
  - 다양성 보장 (같은 장르/아티스트 연속 방지)
  - 비즈니스 규칙 (광고, 프로모션 삽입)
  - 신선도 (오래된 콘텐츠 제거)
  - 필터링 (이미 본 것, 부적절한 콘텐츠 제거)
  - 공정성 (특정 공급자 편중 방지)
```

## 학습 확장성

추천 시점(서빙)뿐 아니라 **학습(트레이닝)**도 확장해야 한다.

### 분산 학습

```
데이터 병렬화:
  데이터를 여러 GPU/머신에 나누어 학습
  → 각 머신이 일부 데이터로 그래디언트 계산
  → 그래디언트를 합쳐서 모델 업데이트

모델 병렬화:
  모델이 너무 크면 여러 GPU에 나누어 올림
  → 임베딩 테이블이 수백 GB일 때 필요
```

### 증분 학습 (Incremental Learning)

```
배치 학습:
  전체 데이터로 매일 밤 재학습 (8시간)
  → 오늘 낮의 트렌드가 내일 아침에야 반영

증분 학습:
  새 데이터가 들어올 때마다 모델을 업데이트
  → 실시간에 가까운 반영
  → 하지만 모델 품질 관리가 어려움

실무:
  배치 학습 (주기적) + 실시간 피처 업데이트 (증분) 병행
```

## 전체 아키텍처 예시

```
┌─────────────────────────────────────────────┐
│                  오프라인                     │
│                                              │
│  학습 데이터 → 모델 학습 → 아이템 임베딩 계산  │
│                           → ANN 인덱스 구축   │
│                                              │
└──────────────────┬──────────────────────────┘
                   │ 모델, 인덱스 배포
                   ↓
┌─────────────────────────────────────────────┐
│                  온라인                       │
│                                              │
│  요청 → 유저 벡터 계산 → ANN 검색 (후보 생성) │
│                        → 랭킹 모델 (점수)     │
│                        → 재정렬               │
│                        → 응답 (10~50개)       │
│                                              │
│  지연 시간 목표: 100ms 이내                   │
└─────────────────────────────────────────────┘
```

## 정리

```
확장성 = 수억 아이템에서 수십 ms 안에 추천하는 것

핵심 아키텍처: 후보 생성 → 랭킹 → 재정렬 (Two-Stage+)

후보 생성:
  Two-Tower Model로 유저/아이템 벡터 독립 계산
  ANN (HNSW, IVF 등)으로 가장 가까운 아이템 빠르게 검색
  여러 소스에서 후보를 모아 다양성 확보

랭킹:
  복잡한 모델 (DeepFM, Transformer 등)로 정밀 점수
  교차 피처, 컨텍스트 등 풍부한 정보 활용

재정렬:
  다양성, 비즈니스 규칙, 공정성 적용

학습:
  분산 학습 + 증분 학습으로 대규모 데이터 처리
```
