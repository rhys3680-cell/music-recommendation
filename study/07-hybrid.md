# 7. 하이브리드 추천 (Hybrid Recommendation)

## 왜 하이브리드인가?

콘텐츠 기반과 협업 필터링은 각각 다른 약점을 가진다.

```
콘텐츠 기반 (CBF):
  ✓ 새 아이템 추천 가능
  ✗ 필터 버블, 다양성 부족
  ✗ 새 사용자에게 추천 불가

협업 필터링 (CF):
  ✓ 속성 없이도 추천 가능
  ✓ 의외의 발견 가능
  ✗ 새 아이템 추천 불가
  ✗ 데이터 희소성에 약함
```

```
핵심 관찰:
CBF가 약한 곳에서 CF가 강하고
CF가 약한 곳에서 CBF가 강하다
→ 둘을 합치면 서로의 약점을 보완할 수 있다
```

---

## 하이브리드 결합 전략

### 1. Weighted (가중 결합)

각 방법의 예측 점수를 **가중 평균**한다. 가장 단순한 방식이다.

```
score(u, i) = α × score_CF(u, i) + (1-α) × score_CBF(u, i)

α = 0.7이면 CF 70%, CBF 30% 반영
```

```
예시:
CF 예측:  영화A = 4.2,  영화B = 3.8
CBF 예측: 영화A = 3.5,  영화B = 4.5
α = 0.6

영화A 최종 = 0.6×4.2 + 0.4×3.5 = 2.52 + 1.40 = 3.92
영화B 최종 = 0.6×3.8 + 0.4×4.5 = 2.28 + 1.80 = 4.08

→ CF만 쓰면 영화A 추천, 하이브리드는 영화B 추천
```

- **장점**: 구현 매우 간단
- **단점**: 최적 가중치를 찾기 어려움, 상황에 따라 달라야 함

### 2. Switching (전환)

상황에 따라 **하나의 방법만 선택**해서 사용한다.

```
if 사용자 이력이 충분하면:
    CF 사용
else:
    CBF 사용 (또는 인기도 기반)
```

```
예시:
유저A (평가 50개) → CF로 추천 (데이터 충분)
유저B (평가 3개)  → CBF로 추천 (CF 신뢰도 낮음)
신규 유저         → 인기 아이템 추천
```

- **장점**: 각 상황에 최적의 방법 적용
- **단점**: 전환 기준 설정이 어려움, 경계에서 추천 품질 불안정

### 3. Cascade (단계적)

한 방법으로 **후보를 줄이고**, 다른 방법으로 **순위를 매긴다**.

```
1단계: CF로 상위 100개 후보 선정 (대략적 필터링)
2단계: CBF로 100개 후보의 순위 재조정 (정밀 랭킹)
```

```
예시:
1단계 CF 결과 (상위 5):
  영화A(4.5), 영화B(4.3), 영화C(4.1), 영화D(4.0), 영화E(3.9)

2단계 CBF로 재정렬:
  영화C(속성 매우 유사) → 1위
  영화A(속성 유사)     → 2위
  영화E(속성 유사)     → 3위
  영화B(속성 보통)     → 4위
  영화D(속성 낮음)     → 5위
```

- **장점**: 각 방법의 장점을 단계적으로 활용
- **단점**: 1단계에서 좋은 아이템이 걸러지면 복구 불가

### 4. Feature Combination (피처 결합)

CBF의 피처와 CF의 피처를 **하나의 모델 입력으로 합쳐서** 학습한다.

```
입력 피처 = [CF 잠재 벡터] + [아이템 속성] + [사용자 프로필]

예시:
유저A-영화X 피처 벡터:
  CF 잠재 벡터:   [0.2, 1.5, 0.8]        (MF에서 학습)
  아이템 속성:     [액션=1, 로맨스=0, ...]  (콘텐츠)
  사용자 프로필:   [평균평점=4.2, 활동량=높음]

→ 하나의 모델 (로지스틱 회귀, 신경망 등)에 입력
→ 클릭/구매 확률 예측
```

- **장점**: 모든 정보를 통합하여 최적화 가능
- **단점**: 모델 복잡도 증가, 피처 엔지니어링 필요

### 5. Meta-Level (메타 수준)

한 방법의 **출력을 다른 방법의 입력**으로 사용한다.

```
1. CBF로 아이템 프로필 벡터 생성
2. 이 벡터를 CF의 아이템 표현으로 사용

또는:

1. CF로 사용자 잠재 벡터 학습
2. 이 벡터를 CBF의 사용자 프로필로 사용
```

- **장점**: 두 방법이 유기적으로 연결
- **단점**: 의존성이 생겨 한쪽이 나쁘면 전체가 나빠짐

## 전략 비교 요약

| 전략 | 구현 난이도 | 유연성 | Cold Start 대응 | 실무 활용 |
|------|-----------|--------|----------------|----------|
| Weighted | 쉬움 | 낮음 | 보통 | 프로토타입 |
| Switching | 보통 | 높음 | **좋음** | 초기 서비스 |
| Cascade | 보통 | 보통 | 보통 | 검색/추천 |
| Feature Combination | 어려움 | **높음** | **좋음** | **대규모 서비스** |
| Meta-Level | 어려움 | 보통 | 보통 | 연구 |

## 실제 서비스 사례

### Netflix

```
1단계: 여러 알고리즘으로 후보 생성 (CF, CBF, 인기도 등)
2단계: 각 알고리즘의 결과를 피처로 합침
3단계: 메타 모델(앙상블)로 최종 순위 결정

→ Cascade + Feature Combination 혼합
```

### YouTube

```
1단계: Candidate Generation (후보 생성)
  - CF 기반으로 수백만 영상 → 수백 개로 축소

2단계: Ranking (랭킹)
  - 사용자 피처 + 영상 피처 + 컨텍스트를 결합한 딥러닝 모델
  - 시청 확률 예측

→ Cascade + Feature Combination
```

### Spotify

```
CF:  "비슷한 사용자가 들은 곡"
CBF: "오디오 특성이 비슷한 곡"
NLP: "비슷한 태그/가사의 곡"

→ 여러 시그널을 결합하여 Discover Weekly 등 생성
```

## Cold Start에 대한 하이브리드의 해결

```
상황별 대응:

새 사용자 + 새 아이템:
  → 인기도 기반 추천 (어떤 CF/CBF도 어려움)

새 사용자 + 기존 아이템:
  → CBF 불가 (프로필 없음), CF 불가 (이력 없음)
  → 가입 시 취향 조사로 초기 프로필 생성 → CBF 시작
  → 몇 개 평가 후 CF 전환 (Switching)

기존 사용자 + 새 아이템:
  → CF 불가 (아무도 평가 안 함)
  → CBF로 아이템 속성 기반 추천 가능
  → 사용자에게 노출 후 데이터 쌓이면 CF 합류
```

## 정리

```
하이브리드 추천 = CBF와 CF의 약점을 서로 보완하는 결합 전략

주요 전략:
  Weighted:  가중 평균 (가장 단순)
  Switching: 상황에 따라 전환 (Cold Start 대응)
  Cascade:   후보 생성 → 랭킹 (실무 표준)
  Feature Combination: 피처 통합 (가장 강력)

실무에서는:
  Cascade + Feature Combination 조합이 표준
  → 후보 생성 (CF) → 랭킹 (통합 모델)

이것으로 Phase 2 핵심 알고리즘이 끝났다.
→ Phase 3에서는 암시적 피드백, 딥러닝, 시퀀스 모델 등 심화로 넘어간다.
```
